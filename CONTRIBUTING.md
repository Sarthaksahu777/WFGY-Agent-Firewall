# CONTRIBUTING

## Phase 1 rule

Keep changes small, narrow, and reviewable.

This repository is in an MVP sprint.
Avoid broad refactors, framework swaps, or speculative architecture changes unless discussed first.

## Workflow

1. pick an issue
2. create a branch or fork
3. keep the PR focused
4. add a short test note or demo note if relevant
5. open the PR for review

## PR preference

Good Phase 1 PRs usually do one thing only:

- wire one hook
- add one critic
- add one logging unit
- add one demo
- improve one config file

## Avoid for now

- large renames
- broad restructuring
- adding Phase 2 systems into Phase 1 PRs
- unrelated cleanup PRs
